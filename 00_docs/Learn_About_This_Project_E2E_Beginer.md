# WEAV AI: 응애 나 애기 주니어 개발자 프로젝트 이해 가이드 (7일)

이 문서는 “응애 나 애기 주니어 개발자”가 **WEAV AI 프로젝트를 이해하고 팀과 같은 언어로 대화**할 수 있도록 돕는 문서입니다.  
목표는 코드를 짜는 게 아니라, **이 프로젝트가 어떻게 돌아가는지**를 “프로젝트 기준으로” 설명할 수 있게 만드는 것입니다.

---

## 0. 먼저 결론(이 프로젝트가 뭘 하는 서비스인가)

WEAV AI는 크게 3개의 기능이 한 앱 안에 들어있는 서비스입니다.

1) **채팅**: 사용자가 질문을 쓰면 AI가 답해주는 채팅  
2) **이미지 생성**: 사용자가 설명(프롬프트)을 쓰면 AI가 이미지를 만들어주는 기능  
3) **문서 기반 질문(RAG)**: PDF 같은 문서를 올리고, 그 문서에서 근거를 찾아 답해주는 기능(답변 근거를 PDF에서 하이라이트로 보여줌)

그리고 “WEAV Studio”라는 추가 기능이 있습니다.

4) **WEAV Studio**: 유튜브 기획/트렌딩/벤치마킹/대본/이미지/TTS(음성)/영상 export/메타데이터/썸네일 제작을 도와주는 “워크플로우 화면”

---

## 2. Day 1 - IT 기초를 WEAV AI로만 배우기

### 2.1 “프로젝트”와 “저장소(repo)”는 뭐야?

- **프로젝트**: 한 제품(WEAV AI)을 만들기 위한 “파일들의 묶음”
- **저장소(repo)**: 그 파일 묶음을 Git이 관리하는 단위(버전 기록이 남음)

WEAV AI 저장소에는 크게 이런 폴더가 있습니다.

- `frontend/`: 화면(UI)
- `backend/`: 서버(API)
- `infra/`: Docker로 서버를 띄우는 설정(데이터베이스/Redis/MinIO 포함)
- `00_docs/`: 문서

### 2.1.1 Docker는 뭐야? (WEAV AI에서 왜 쓰는지)

WEAV AI는 버튼 한 번으로 “서버/DB/저장소”를 한꺼번에 띄우기 위해 **Docker**를 씁니다.

- Docker를 아주 쉽게 말하면: “프로그램을 **정해진 환경(상자)** 안에 넣어서 실행하는 방식”
- 그 상자를 **컨테이너(container)** 라고 부릅니다.

WEAV AI는 Docker로 아래를 한꺼번에 올립니다.

- 백엔드 서버(Django API)
- DB(PostgreSQL)
- 작업 큐/비동기용(Redis)
- 파일 저장소(MinIO)
- 앞단 라우팅(Nginx)

왜 Docker를 쓰나(기획자 관점 요약):

- 팀원이 각자 PC 환경이 달라도 “같은 방식으로” 실행되기 쉬움
- “서버/DB/저장소”를 따로 설치하지 않고 한 번에 올릴 수 있음

### 2.1.2 포트(port)와 주소(URL)는 뭐야? (WEAV AI 기준)

“주소”는 접속 위치입니다. 예: `http://localhost:8080`

- `localhost`: 내 컴퓨터를 의미
- `8080`, `3001` 같은 숫자: 같은 컴퓨터 안에서 “어떤 프로그램으로 들어갈지”를 구분하는 문(포트)

WEAV AI에서 자주 쓰는 주소:

- 백엔드(서버) 확인: `http://localhost:8080/api/v1/health/`
- 프론트(UI) 화면: `http://localhost:3001`

### 2.2 Git은 뭐야? (아주 간단히)

Git은 “파일 변경 이력”을 남기는 도구입니다.

- 누가 언제 어떤 파일을 어떻게 바꿨는지 기록
- 되돌리기 가능
- 팀이 동시에 작업하기 가능

기획자 입장에서는 “작업 기록이 남고 협업이 되는 시스템” 정도로 이해하면 충분합니다.

### 2.3 프레임워크(framework)는 뭐야?

프레임워크는 “제품을 빨리 만들게 해주는 기본 뼈대/도구 세트”입니다.

WEAV AI의 프레임워크 예시:

- 프론트: **React** (화면을 만드는 뼈대)
- 백엔드: **Django** (서버를 만드는 뼈대)

비유:

- 프레임워크는 “레고 기본 세트”
- 우리는 그 위에 “WEAV AI만의 부품과 규칙”을 추가해 제품을 만듭니다.

## 3. Day 2 - 프론트/백엔드/API를 “화면 흐름”으로 이해하기

### 3.1 프론트엔드(frontend)는 뭐야?

**프론트엔드**는 사용자가 직접 보는 화면(UI)입니다.

WEAV AI에서 프론트가 하는 일:

- 채팅 입력창을 보여준다
- “전송” 버튼을 누르면 서버에 요청한다
- 서버에서 온 답변을 화면에 보여준다
- PDF를 화면에 렌더링하고, “답변 근거”를 하이라이트로 표시한다
- 텍스트 프롬프트 조립기와 이미지 프롬프트 생성기로 입력 초안을 만들어준다

### 3.2 백엔드(backend)는 뭐야?

**백엔드**는 “요청을 받아서 처리하고 결과를 주는 서버”입니다.

WEAV AI 백엔드가 하는 일:

- 채팅 요청이 오면 AI에게 물어보고 답을 만든다
- 이미지 생성 요청이 오면 AI에게 이미지를 만들게 한다
- 사용자가 올린 문서를 저장하고(업로드), 문서 내용을 “검색 가능한 형태”로 바꾼다(RAG)

### 3.3 API는 뭐야?

**API**는 프론트가 백엔드에 “무엇을 해달라”고 말하는 규칙(약속)입니다.

예시(WEAV AI):

- 프론트가 백엔드에 “이 질문에 답해줘” 요청을 보냄
- 백엔드는 “알겠어, 지금 처리 중이야”라고 먼저 답함(202)
- 처리가 끝나면 프론트가 결과를 다시 가져와서 화면에 보여줌

### 3.4 HTTP/요청/응답을 간단히

- 요청(request): “주세요/해주세요”
- 응답(response): “여기요/결과입니다”

WEAV AI에서는 요청/응답이 웹(브라우저)로 오가고, JSON이라는 형식(텍스트)로 많이 주고받습니다.

## 4. Day 3 - DB/저장/데이터를 WEAV AI 데이터로 이해하기

### 4.1 DB(데이터베이스)는 뭐야?

DB는 “데이터를 저장하는 창고”입니다.

WEAV AI는 DB에 이런 걸 저장합니다.

- 어떤 채팅방(세션)이 있었는지
- 누가 어떤 메시지를 보냈는지
- 어떤 문서를 업로드했는지, 처리 상태는 어떤지
- 어떤 이미지가 생성됐는지

### 4.2 SQL이 뭐야? NoSQL이 뭐야? (정확한 정의보다 감각)

이 프로젝트에서 중요한 건 “WEAV AI는 PostgreSQL을 쓴다”입니다.

- **SQL DB(PostgreSQL)**: 표(테이블) 형태로 깔끔하게 저장하고, 검색/정렬/조건 조회가 강함
- **NoSQL**: 표보다는 “문서(JSON)” 형태로 유연하게 저장하는 방식이 많음

WEAV AI는 “채팅방/메시지/문서/작업” 같은 데이터가 관계를 가지기 때문에 SQL(PostgreSQL)이 잘 맞습니다.

### 4.3 WEAV AI에서 “세션(Session)”이 뭐야?

세션(Session)은 “한 작업 공간”이라고 이해하면 됩니다.

- 채팅 세션: 대화가 저장되는 방
- 이미지 세션: 이미지 생성 기록이 저장되는 방
- 스튜디오 세션: 스튜디오 프로젝트 목록에 보이는 단위(단, 상세 진행상태는 브라우저에 저장)

### 4.4 WEAV AI에서 “문서(Document)”는 뭐야?

문서는 “세션에 업로드한 파일”입니다(PDF/HWP/HWPX).

문서가 업로드되면:

- “처리 중” 상태가 되고
- 문서 내용이 쪼개져서(청크)
- 검색 가능하도록 준비됩니다(RAG)

## 5. Day 4 - 비동기(Async)를 WEAV AI의 ‘기다림’으로 이해하기

### 5.1 비동기(Async)는 뭐야?

비동기는 “시간이 오래 걸리는 일을, 지금 당장 화면을 멈추지 않고 뒤에서 처리하는 방식”입니다.

WEAV AI에서 오래 걸리는 일:

- AI 채팅 답변 만들기
- 이미지 생성
- 문서 분석/검색 인덱스 만들기

### 5.2 비동기가 없으면 어떤 문제가 생겨?

만약 비동기가 없으면:

- 버튼 누르고 화면이 오래 멈춤
- “응답 시간 초과”가 나기 쉬움
- 사용자 경험(UX)이 나빠짐

### 5.3 WEAV AI는 비동기를 위해 뭘 쓰나?

WEAV AI는 이런 조합을 씁니다.

- **Celery**: “뒤에서 일할 작업자(워커)” 시스템
- **Redis**: “작업 주문서(큐)를 저장하는 창구”

비유:

- 프론트(손님) -> 백엔드(카운터) -> 작업 큐(주문서) -> 워커(주방) -> 결과(DB/스토리지)

### 5.4 Job(작업 상태)는 뭐야?

WEAV AI에는 “Job”이라는 개념이 있습니다.

- 채팅을 보내면 “지금 처리 중” Job이 하나 생김
- 이미지 생성도 마찬가지
- 프론트는 Job 상태를 주기적으로 확인(polling)해서, 완료되면 결과를 화면에 표시

## 6. Day 5 - 문서 RAG를 “근거 하이라이트”로 이해하기

### 6.1 RAG는 뭐야? (가장 쉬운 설명)

RAG는 “AI가 아무 말(환각)을 하지 않게, **문서에서 근거를 찾아** 답하게 만드는 방식”입니다.

### 6.2 WEAV AI의 문서 RAG는 어떻게 보이나?

사용자 경험:

1) 채팅방에서 문서를 업로드한다
2) 문서 처리(status)가 완료될 때까지 기다린다
3) 질문할 때 문서명을 `@문서명`으로 지정한다  
4) AI가 답을 할 때 “답변 근거 확인” 버튼을 누르면
5) PDF 화면에서 근거 부분이 하이라이트된다

여기서 중요한 건 “근거가 화면에서 보인다”입니다.  
기획 관점으로는 “설명 가능성/신뢰”를 만드는 장치입니다.

### 6.3 fallback(폴백)은 뭐야?

fallback은 “원래 방법이 실패하면, 대체 방법으로 넘어가는 것”입니다.

WEAV AI 예시:

- 문서에서 텍스트가 잘 추출되지 않으면 OCR(이미지에서 글자 읽기)로 대체할 수 있음
- Studio에서 유튜브 직접 영상 분석이 실패하면 메타데이터/자막 기반 분석으로 넘어감

## 7. Day 6 - WEAV Studio를 “제작 워크플로우”로 이해하기

WEAV Studio는 “유튜브 콘텐츠 제작” 흐름을 한 화면에서 돕는 도구입니다.

Studio에서 하는 대표 행동:

- 트렌딩 영상 조회(YouTube 데이터)
- 유튜브 URL을 넣고 벤치마킹 분석(요약/패턴)
- 주제/대본/장면을 AI로 생성
- 장면 이미지와 장면용 이미지 프롬프트를 생성
- TTS(음성)를 생성
- 최종 영상을 export하고, 제목/설명/고정댓글을 생성
- 썸네일 벤치마킹 후보를 만들고 완성 미리보기를 확인

중요한 특징(기획자 관점):

- Studio는 DB에 모든 중간 데이터를 저장하기보다, **브라우저(localStorage)** 에 많이 저장합니다.
- 이유(쉽게): “빠르게 만들고 빠르게 실험하려고”
- 단점: 다른 PC/브라우저에서 이어하기가 어려움(기획/운영에서 알고 있어야 함)

## 9. 부록 - 이 문서가 다루지 않는 것

이 문서는 “WEAV AI를 이해하기 위한 최소한의 IT 기초”만 다룹니다.

- 코딩 문법(파이썬/자바스크립트) 자체

필요해지면 그때 “WEAV AI 기준”으로만 추가 문서를 만드는 게 가장 빠릅니다.
